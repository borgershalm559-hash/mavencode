# Дизайн: Генератор блок-схем для MavenCode

**Дата:** 2026-05-03  
**Проект:** MavenCode  
**Цель:** Создание блок-схем процессов в формате JSON (совместимом с diagram (3).json)

## Обзор

Создать набор блок-схем для образовательной платформы MavenCode, отражающих реальную логику процессов на основе анализа исходного кода. Диаграммы будут в том же JSON-формате, что и референсная диаграмма diagram (3).json.

## Требования

### Функциональные требования

1. **Анализ кода проекта**
   - Сканирование API routes в `src/app/api/*`
   - Анализ auth логики в `src/lib/auth.ts` и `src/lib/auth.config.ts`
   - Изучение Prisma схемы для понимания моделей данных
   - Извлечение валидаций, условий, обработки ошибок

2. **Генерация диаграмм для процессов**
   
   **Пользовательские процессы (A):**
   - Регистрация пользователя (student/teacher)
   - Вход в систему
   - Прохождение урока
   - Получение достижений и XP
   
   **Административные процессы (B):**
   - Создание/редактирование курса
   - Управление уроками
   - Модерация пользователей
   - Публикация новостей

3. **Формат выходных данных**
   - JSON-файлы в формате diagram (3).json
   - Типы блоков: "Начало / конец", "Блок", "Ввод / вывод", "Условие", "Текст / комментарий"
   - Средний уровень детализации: ключевые шаги + основные проверки, без избыточности

### Нефункциональные требования

1. **Точность:** Диаграммы должны отражать реальную логику из кода
2. **Читаемость:** Блоки должны быть расположены логично, связи понятны
3. **Совместимость:** JSON должен открываться в той же программе, что и diagram (3).json
4. **Полнота:** Охват всех ключевых процессов пользователей и администраторов

## Архитектура решения

### Компоненты

1. **Code Analyzer**
   - Читает и парсит исходный код
   - Извлекает логику процессов, условия, валидации
   - Строит промежуточное представление процесса

2. **Process Graph Builder**
   - Преобразует логику в граф узлов и рёбер
   - Определяет типы блоков
   - Выстраивает последовательность шагов

3. **Layout Engine**
   - Вычисляет координаты блоков (x, y)
   - Обеспечивает читаемое расположение
   - Избегает пересечений стрелок где возможно

4. **JSON Generator**
   - Конвертирует граф в формат diagram (3).json
   - Создаёт структуру blocks и arrows
   - Сохраняет файлы

### Структура JSON-формата

```json
{
  "blocks": [
    {
      "x": 260,
      "y": -20,
      "text": "начало",
      "width": 100,
      "height": 30,
      "type": "Начало / конец",
      "isMenuBlock": false,
      "fontSize": 14,
      "textHeight": 14,
      "isBold": false,
      "isItalic": false,
      "textAlign": "center",
      "labelsPosition": 1
    }
  ],
  "arrows": [
    {
      "startIndex": 0,
      "endIndex": 1,
      "startT": 1,
      "endT": 0,
      "mode": 0,
      "labelMode": 0,
      "label": ""
    }
  ]
}
```

### Типы блоков

- **"Начало / конец"** - начальная и конечная точки процесса
- **"Блок"** - действие, операция
- **"Ввод / вывод"** - ввод данных пользователем или вывод результата
- **"Условие"** - проверка условия, ветвление
- **"Текст / комментарий"** - пояснения, комментарии

## Процессы для диаграмм

### 1. Регистрация пользователя

**Файл:** `mavencode-registration.json`

**Ключевые шаги:**
- Начало
- Выбор роли (student/teacher)
- Ввод данных (email, пароль, ФИО, телефон)
- Для student: класс
- Для teacher: код учителя
- Проверка email на уникальность
- Валидация данных
- Хеширование пароля
- Создание записи в БД
- Отправка письма подтверждения (опционально)
- Конец

**Источники кода:**
- `src/app/api/register/route.ts`
- `src/lib/auth.ts`

## Расположение блоков (Layout Strategy)

### Принципы позиционирования

1. **Вертикальный flow:** Основной поток сверху вниз
2. **Горизонтальное ветвление:** Условия создают горизонтальные ветки
3. **Отступы:**
   - Между блоками по вертикали: 80px
   - Между блоками по горизонтали: 140px
   - Начальная позиция: x=260, y=-20

4. **Размеры блоков:**
   - Начало/конец: 100x30
   - Блок: 100x40
   - Ввод/вывод: 120x40 (или больше для многострочного текста)
   - Условие: 100x60
   - Текст: динамический размер

## Ограничения и допущения

1. **Средний уровень детализации:** Не включаем каждую мелкую проверку, фокусируемся на ключевых шагах
2. **Обработка ошибок:** Включаем основные error paths, но не все edge cases
3. **Async операции:** Упрощаем асинхронные операции до синхронного представления
4. **UI взаимодействия:** Фокус на бизнес-логике, а не на UI-деталях
